## 1. ホーミング弾の「距離依存ターンアシスト」

### 実現したかったこと

ジャストエヴェイジョン成立時に発射されるカウンター弾には、2 つの相反する要件があった。

- **視覚的インパクト**：直進ではなく曲線軌道で派手に飛んでほしい
- **確実なヒット保証**：爽快感の核心であるカウンターが「外れる」とゲームとして成立しない

### 比較検討した手法

| 案 | 内容 | 問題点 |
|---|---|---|
| A | 直進弾 | 見た目が地味で演出にならない |
| B | 単純ホーミング（一定ターン速度） | 遠距離でも不自然に曲がる、軌道が単調 |
| C | サイン波軌道のみ | 近距離でカオスがキャンセルしきれず外れることがある |
| **D（採用）** | サイン波＋距離依存ターンアシスト | 遠距離は派手、近距離は確実 |

### 技術選定の理由

C 案（サイン波）単体では「外れる不安」が残った。プロファイルを取ったわけではなく、プレイテストの感触として「近距離での外れ」が不快であることが判明。  
距離に応じてカオス振幅を減衰させ、ターン速度を強化する仕組みを追加することで、**「遠距離では見た目重視・近距離では当たり重視」** という 2 フェーズ設計に落ち着いた。

### 実装（実コード抜粋）

```cpp
// PlayerCircusBullet.cpp
if (distanceToTarget < 6.0f) {
    float t = distanceToTarget / 6.0f;  // 0→1（接近につれ0へ）
    playerChaosAmp  *= t;               // カオス振幅を距離比でフェードアウト
    playerTurnSpeed *= (2.0f - t);      // ターン速度を最大 2x まで強化
}
```

サイン波の生成には軸ごとに異なる周波数倍率（1.0x / 1.2x / 0.8x）とランダムな位相オフセットを使い、非均一な螺旋軌道を実現した。

### 定量的な結果

| パラメータ | 値 |
|---|---|
| ターン速度ブースト上限 | **2.2 倍** |
| カオス振幅の抑制率（プレイヤー弾） | **40%**（敵弾の 60% 比） |
| 距離切り替え閾値 | **6 ユニット** |
| フェーズ 1（減速）期間 | **0.4 秒** |

### トレードオフ

プレイヤー弾は敵弾より意図的に「当たりやすく」設計されている（非対称バランス）。リアリズムは捨てたが、ゲームとして正しい判断。難易度調整をパラメータ化しているため、将来的な変更にも対応できる。

## 2. ジャストエヴェイジョン — 入力ウィンドウ設計

### 実現したかったこと

格闘ゲームのジャストガードと同様の「ギリギリ感」を出しつつ、カジュアルユーザーにも成功体験を与えたい。  
かつ、フレームレートが変動しても判定がブレない実装が必要だった。

### 比較検討した手法

| 案 | 問題点 |
|---|---|
| A. 弾との距離判定のみ | 弾速が変わるたびに閾値の再調整が必要 |
| B. フレーム数カウント | 60fps 固定前提になりフレームレート非依存にならない |
| **C. 秒数ウィンドウ（採用）** | フレームレート非依存、パラメータが直感的で調整しやすい |

### 実装（実コード抜粋）

```cpp
// PlayerEvasion.cpp
if (isDodging_ && justEvasionTimer_ <= justEvasionWindow_ && !hasJustEvaded_) {
    hasJustEvaded_ = true;
    return true;  // ジャスト成立
}
```

- `isDodging_` が true の間、毎フレーム判定を行う
- `hasJustEvaded_` フラグで「1 回限り発火」を保証
- タイマーは回避開始から加算（弾との距離判定ではない）

### 定量的な結果

| パラメータ | 値 |
|---|---|
| ジャスト入力受付ウィンドウ | **3.0 秒**（`justEvasionWindow_`） |
| 回避継続時間 | **0.2 秒**（`dodgeDuration_`） |
| 回避クールダウン | **1.0 秒** |

### トレードオフ

ウィンドウを 3.0 秒と広めにとることで初心者でも成功体験が得やすい。上級者向けには値を絞るだけで難易度調整が可能。「何フレーム」ではなく「何秒」で管理することで、デザイナーが直感的にパラメータを触れる設計にした。

## 3. DirectX 12 SRV デスクリプタヒープ — 線形アロケーション

### 実現したかったこと

DirectX 12 では SRV（テクスチャ・構造化バッファ等）を自前のデスクリプタヒープで管理する必要がある。確保・解放を効率よく行う仕組みが必要だった。

### 比較検討した手法

| 案 | 内容 | 問題点 |
|---|---|---|
| A | フリーリスト | 実装が複雑、断片化リスクあり、このゲームには過剰 |
| B | プールアロケーター | 断片化対策はできるが同様に実装コストが高い |
| **C（採用）** | 線形アロケーション | O(1) 確保、実装がシンプル、起動時ロードのみの要件に十分 |

### 技術選定の理由

このゲームはアセットを**起動時に全ロードし、ゲーム中に動的ロード/アンロードは行わない**。  
フリーリストが必要になる「動的な入れ替え」が発生しないため、線形アロケーションで十分と判断した。  
「汎用的な正しさ」より「要件にフィットしたシンプルな設計」を優先した。

### 実装（実コード抜粋）

```cpp
// SrvManager.cpp
uint32_t SrvManager::Allocate() {
    uint32_t index = useIndex;
    useIndex++;       // インクリメントのみ、O(1)
    return index;
}
```

### 定量的な結果

| パラメータ | 値 |
|---|---|
| 最大 SRV スロット数 | **1024** |
| 確保コスト | **O(1)**（インクリメントのみ） |

### トレードオフ

解放処理が存在しない（一度確保したインデックスは戻せない）。シーン中にテクスチャを動的に差し替えるゲームには不向き。今回は起動時ロードのみなので問題ないが、将来的にダイナミックロードを実装する場合はフリーリストへの移行が必要になる。

## 4. ビヘイビアツリー — 関数合成による軽量実装

### 実現したかったこと

複数種の敵（ラッシュ、モルタル、ボス）が「索敵 → 追跡 → 攻撃」という共通の行動パターンを持ちつつ、それぞれ異なる攻撃を実行できる AI 基盤が必要だった。

### 比較検討した手法

| 案 | 問題点 |
|---|---|
| A. 有限ステートマシン | 状態数が増えると遷移条件の組み合わせが爆発する |
| B. 既存 BT ライブラリ | 今回の規模に対して依存コストが高い |
| **C. 関数合成 BT（採用）** | ヘッダ ~180 行で実現、ラムダ渡しで各敵が柔軟にツリーを構築 |

### 実装（実コード抜粋）

```cpp
// BehaviorTree.h（概念を示す擬似コード）
// Sequence: 全子が Success なら Success（AND ノード）
// Selector: 最初の Success で打ち切り（OR ノード）
// Condition: bool ラムダ → Success/Failure
// Action:    処理ラムダ → Success/Failure/Running（複数フレーム対応）
```

視野コーン判定は「距離チェック → 角度チェック → レイサンプリング」の順に処理し、早期リターンでコストを最小化。

### 定量的な結果

| パラメータ | 値 |
|---|---|
| 視野角 | **90°** |
| 視野距離 | **50 ユニット** |
| レイサンプリング間隔 | **0.5 ユニット**（壁貫通防止） |
| BT 実装規模 | **ヘッダ ~180 行** |

### トレードオフ

`Running` 状態を持つ ActionNode は複数フレームにまたがる処理に対応するが、コルーチンを使っていないため「どこまで処理が進んだか」を自分で管理する必要がある。シンプルさと引き換えの設計上の制約。

## 5. パーティクルシステム — ファクトリレジストリパターン

### 実現したかったこと

エフェクトの種類（リング、トレイル、オーラ、爆発など 8 種類以上）を増やすたびに `ParticleManager` を修正したくない。新しいエミッター型を追加しても既存コードへの影響をゼロにしたい。

### 比較検討した手法

| 案 | 問題点 |
|---|---|
| A. switch 文による型判別 | 型追加のたびに Manager を修正する必要がある |
| B. 継承＋仮想関数のみ | 型の登録が静的になり、実行時の動的追加が難しい |
| **C. unordered_map ファクトリ（採用）** | 文字列キーで登録・生成、開放/閉鎖の原則に沿う |

### 実装（実コード抜粋）

```cpp
// ParticleManager.cpp（概念を示す実コード）
// 登録
emitterFactories_["Ring"] = []() { return std::make_unique<RingEmitter>(); };

// 生成（文字列キーのみで型を特定）
auto emitter = emitterFactories_[typeName]();
```

バッファ容量が変更された場合は `ReallocateInstanceBufferIfNeeded()` で自動的に GPU バッファを再確保し、超過分のパーティクルをトリミングする。

### 定量的な結果

| パラメータ | 値 |
|---|---|
| エミッター最大インスタンス数 | **500,000** |
| デフォルト最大数 | **128** |
| ImGui アンドゥ履歴深度 | **50 エントリ** |

### 他職種の視点（拡張性）

文字列キーによる登録・生成なので、将来的にプランナーがエフェクト設定を JSON で記述しエンジンに渡すデータ駆動設計へ移行できる構造になっている。

## 6. シーン遷移アニメーション — 対角線ディレイ＋重み付きジッタ

### 実現したかったこと

シーン間の遷移を「ブロックが崩れていくような演出」で表現したい。全ブロックが一斉に消えると機械的すぎるが、完全ランダムだと「波」の流れが失われる。

### 技術選定の理由

「左上から右下への対角線方向に波を作りつつ、乱数でバラつきを加える」ことで有機的な印象を実現。  
さらに後半ブロック（対角線の末端）では乱数の影響を小さくすることで、**波の「まとまり」が崩れない**工夫をした。

### 実装（実コード抜粋）

```cpp
// SceneChangeAnimation.cpp
float progress     = (x + y) / maxIndexSum;        // 左上=0、右下=1
float randomOffset = (rng % 1000) / 1000.0f * kDelayRandomJitterMax;

// 後半ほどジッタの影響を小さくする重み付き
block.delay = duration * kDelayDurationRatio
            * (progress + randomOffset * (1.0f - progress));
```

各ブロックにイーズイン・イーズアウト曲線を適用し、スケール 0.1 → 1.0（出現）または 1.0 → 0.1（消滅）を補間。`alpha == 0` のブロックは描画をスキップしてドローコールを削減。

### 定量的な結果

| パラメータ | 値 |
|---|---|
| フェード割合 | **30%**（全体時間の 30% でフェード完了） |
| ディレイ展開割合 | **70%**（70% の時間でブロックが時差展開） |
| 最大ジッタ | **0.15 秒** |

### トレードオフ

アルゴリズムによる自動生成のため、手付けキーフレームに比べ特定タイミングへの細かい演出制御は難しい。しかしパラメータ（比率・ジッタ量）を変えるだけで雰囲気を調整できるため、反復改善には向いている。

## まとめ：技術選定で一貫した判断軸

このプロジェクトを通じて、以下の判断軸で技術選定を行った。

| 判断軸 | 具体例 |
|---|---|
| **要件に対して十分な解を選ぶ** | SRV の線形アロケーション（起動時ロード専用だからフリーリスト不要） |
| **ゲームフィールを定量的に検証する** | ターンアシストの切り替え距離を 6u に設定し、プレイテストで確認 |
| **汎用性よりシンプルさ** | BT の自作（180 行）、過剰な外部ライブラリを避けた |
| **デザイナー・非エンジニアへの配慮** | 秒数ベースのウィンドウ、パラメータの外部化 |
